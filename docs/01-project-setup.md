# 01 — Project Setup

This document covers bootstrapping the ToyBox PWA Engine from scratch: the directory structure, entry point files, PWA manifest, and initial tooling choices.

---

## Prerequisites

No build tool is strictly required for development, but the following improve the experience:

| Tool | Purpose | Required? |
|------|---------|-----------|
| A modern browser (Chrome/Edge/Safari) | Running and testing the PWA | ✅ Yes |
| A local HTTP server (`npx serve`, `live-server`, or Python) | Service Workers require HTTPS or localhost | ✅ Yes |
| Node.js (v18+) | Optional: for any future bundling or scripts | ⬜ Optional |
| VS Code + ESLint | Code editing with linting | ⬜ Optional |

> **Why no build tool?** ToyBox is intentionally buildless. ES Modules in modern browsers eliminate the need for Webpack/Vite. This keeps the project accessible for contributors and easier for AI models to generate correct game scripts.

## Directory Scaffold

Run the following commands to set up the full directory tree:

```bash
mkdir -p toybox/{assets/{sprites,audio,icons},games,lib}
touch toybox/index.html
touch toybox/styles.css
touch toybox/app.js
touch toybox/sw.js
touch toybox/manifest.json
touch toybox/engine/engine.js
touch toybox/engine/loader.js
touch toybox/engine/audio.js
touch toybox/engine/input.js
touch toybox/engine/storage.js
touch toybox/games/manifest.json
touch toybox/games/memory_match.js
```

### Final Structure

```
toybox/
├── index.html              ← App shell entry point
├── styles.css              ← Global styles + CSS design tokens
├── app.js                  ← Host application (launcher logic)
├── sw.js                   ← Service Worker (caching strategy)
├── manifest.json           ← PWA web app manifest
│
├── engine/
│   ├── engine.js           ← Engine bridge factory (the `engine` object)
│   ├── loader.js           ← Dynamic game script loader
│   ├── audio.js            ← Audio subsystem
│   ├── input.js            ← Touch/pointer input translation
│   └── storage.js          ← IndexedDB wrapper
│
├── games/
│   ├── manifest.json       ← Game catalogue (list of all games)
│   └── memory_match.js     ← First bundled mini-game
│
├── assets/
│   ├── sprites/            ← Sprite sheets and individual PNGs
│   ├── audio/              ← Sound effect files (.ogg / .mp3)
│   └── icons/              ← PWA icons (192x192, 512x512)
│
└── lib/
    └── pixi.min.js         ← PixiJS vendored locally (no CDN needed offline)
```

## Step 1 — `index.html`

The HTML is minimal. It acts purely as an application shell — all visual structure is generated by JavaScript.

```html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />
  <meta name="viewport" content="width=device-width, initial-scale=1.0, user-scalable=no" />
  <meta name="description" content="ToyBox — A fun mini-game console for kids" />
  <meta name="theme-color" content="#1a1a2e" />

  <!-- PWA meta tags -->
  <meta name="apple-mobile-web-app-capable" content="yes" />
  <meta name="apple-mobile-web-app-status-bar-style" content="black-translucent" />
  <link rel="apple-touch-icon" href="/assets/icons/icon-192.png" />

  <link rel="manifest" href="/manifest.json" />
  <link rel="stylesheet" href="/styles.css" />
  <title>ToyBox</title>
</head>
<body>

  <!-- Host UI Layer -->
  <div id="app-shell">
    <!-- Rendered by app.js -->
  </div>

  <!-- Runtime Canvas Layer (hidden by default) -->
  <canvas id="game-canvas" style="display:none;"></canvas>

  <!-- System overlay: back button, volume, settings -->
  <div id="system-overlay" class="hidden">
    <button id="btn-exit-game" aria-label="Exit game">✕</button>
  </div>

  <!-- PixiJS (vendored locally) -->
  <script src="/lib/pixi.min.js"></script>
  <!-- Host application bootstrap -->
  <script type="module" src="/app.js"></script>
</body>
</html>
```

**Key decisions:**
- `user-scalable=no` prevents accidental pinch-zoom while playing
- `<canvas id="game-canvas">` starts hidden; the host shows/hides it on game boot/exit
- `system-overlay` is always rendered on top via `z-index` so the exit button is always accessible
- PixiJS is loaded as a classic `<script>` (not a module) to expose the global `PIXI` object, which game scripts can reference

## Step 2 — `manifest.json` (PWA Manifest)

```json
{
  "name": "ToyBox",
  "short_name": "ToyBox",
  "description": "A fun offline mini-game console for kids",
  "start_url": "/",
  "display": "fullscreen",
  "orientation": "landscape",
  "background_color": "#1a1a2e",
  "theme_color": "#1a1a2e",
  "icons": [
    {
      "src": "/assets/icons/icon-192.png",
      "sizes": "192x192",
      "type": "image/png",
      "purpose": "any maskable"
    },
    {
      "src": "/assets/icons/icon-512.png",
      "sizes": "512x512",
      "type": "image/png",
      "purpose": "any maskable"
    }
  ],
  "categories": ["games", "kids", "entertainment"]
}
```

**Key decisions:**
- `"display": "fullscreen"` removes all browser UI chrome when installed — essential for the console feel
- `"orientation": "landscape"` locks the tablet to landscape mode
- `"purpose": "any maskable"` allows the icon to be shaped by the OS (rounded square on Android, etc.)

## Step 3 — Register the Service Worker

In `app.js`, the very first action is to register the Service Worker:

```javascript
// app.js — top of file
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () => {
    navigator.serviceWorker.register('/sw.js')
      .then(reg => console.log('[ToyBox] SW registered:', reg.scope))
      .catch(err => console.warn('[ToyBox] SW failed:', err));
  });
}
```

The full Service Worker implementation is covered in [06 — Service Worker & Storage](./06-service-worker-caching.md).

## Step 4 — Vendor PixiJS Locally

Because ToyBox must work offline, PixiJS must be bundled into the project rather than loaded from a CDN.

1. Download the latest PixiJS minified bundle from [pixijs.com](https://pixijs.com):
   ```bash
   curl -L https://pixijs.download/release/pixi.min.js -o toybox/lib/pixi.min.js
   ```
2. Add `/lib/pixi.min.js` to the Service Worker's precache list (see [06 — Service Worker & Storage](./06-service-worker-caching.md))

Alternatively, at build time, use `npm install pixi.js` and copy the dist file to `lib/`.

## Step 5 — Game Catalogue (`games/manifest.json`)

This JSON file is the source of truth for the game launcher. The host reads it on startup.

```json
[
  {
    "id": "memory-match",
    "title": "Animal Matching Puzzle",
    "description": "Flip and match pairs of cute animals!",
    "icon": "ui_icon_cow",
    "scriptPath": "games/memory_match.js",
    "ageRange": "2-6",
    "tags": ["matching", "memory", "animals"]
  }
]
```

Fields:

| Field | Type | Description |
|-------|------|-------------|
| `id` | string | Unique game identifier (slug format) |
| `title` | string | Display name shown in the launcher |
| `description` | string | Short description (max 80 chars) |
| `icon` | string | Asset key for the launcher tile icon sprite |
| `scriptPath` | string | Path to the game JS module file |
| `ageRange` | string | Optional age range label |
| `tags` | string[] | Optional searchable tags |

## Step 6 — Local Development Server

Service Workers require either `localhost` or HTTPS. Use one of these to run ToyBox locally:

```bash
# Option A: npx serve (zero config)
npx serve toybox

# Option B: Python (built into macOS/Linux)
cd toybox && python3 -m http.server 8080

# Option C: VS Code Live Server extension
# Right-click index.html → "Open with Live Server"
```

Open `http://localhost:8080` in Chrome/Edge for the best DevTools experience (Application tab → Service Workers / Cache Storage / IndexedDB).
